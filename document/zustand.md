```zustand```
* 한개의 중앙 집중된 store를 사용하는 구조
* 상태 정의화 사용이 단순하다
* 리덕스와 비교
  * 리덕스와 비교하면 둘다 불변 상태 모델을 기반으로 한다. 
  * 그렇지만 리덕스는 사용하려면 앱을 컨택스트 api 사용해서 래핑 해서 사용해야한다
  * zustand는 그렇지 않음
  * 렌더링 최적화도 리덕스와 접근 방식이 크게 다르지 않다함
* 리코일과 비교(조타이랑 차이점이 비슷함)
  * 앱을 컨택스트 api로 래핑해서 사용해야함
  * 리코일은 atom 의존성?을 통해 렌더 최적화를 하는 방면에, zustand는 selector?를 통해 렌더 최적화를 한다함
    * -> 더 찾아봐야할듯

* 문법들
* state와 set함수만 정의해주면 된다.
* 정의하고 컴포넌트 내 필요한 곳에서 
  * const firstName = userStore((state) => state.firstName); 이런식으로 hook 써서 가져다 쓰면됨
  * 기본적으로 선언한 userStore() 를 호출하면 그 안에있는 상태값과 set 함수들을 모두 다 컴포넌트로 가지고 오게된다
  * 구조분해 할당으로 가져와도 사용을 할 수 있지만, 필요없는 값 때문에 컴포넌트 리 렌더링이 발생할 수도 있다.
  * 위 코드의 (state) => state.firstName 인자로 들어가는 이 함수를 selector 함수라고 한다. 이함수를 통해서 값을 가져오면 리렌더링 없이 가져올수 있다
* deep 하게 중첩된 object 상태는 주로 immer 사용해서 상태 변경해야함
* single zustand store에 상태가 저장됨
  * 앱이 커지면 zustand 문서에 store slice 하는 방법 나옴
